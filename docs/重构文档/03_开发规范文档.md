# StarPath 开发规范文档 v2.0

> 版本：2.0 | 日期：2026-05-10 | 状态：重构基准文档

## 一、总则

### 1.1 文档用途

本文件定义了StarPath项目在v2.0重构及后续迭代中的编码规范、Git工作流、代码审查标准和文档规范。所有贡献者必须遵守。

### 1.2 核心原则

| 原则 | 说明 |
|------|------|
| **类型即文档** | TypeScript类型和Python类型注解必须完整，类型本身就是最好的文档 |
| **文件夹即边界** | 模块（feature）之间通过明确的接口通信，禁止跨模块直接引用内部实现 |
| **测试即信心** | 每个PR必须包含对应测试，无测试不合并 |
| **渐进式实现** | 先建立可用闭环，再优化性能和智能，绝不一次性过度设计 |

## 二、代码规范

### 2.1 命名规范

| 元素 | 规范 | 示例 |
|------|------|------|
| **文件/文件夹** | kebab-case | `guided-tour/`, `task-service.ts` |
| **React组件** | PascalCase | `LoginPage`, `AbilityChart` |
| **React组件文件** | 与组件同名 | `LoginPage.tsx` |
| **Hooks** | camelCase，use前缀 | `useAuth`, `useGuidance` |
| **Store** | camelCase，Store后缀 | `authStore.ts`, `profileStore.ts` |
| **Python类** | PascalCase | `UserService`, `EvaluatorAgent` |
| **Python函数/变量** | snake_case | `get_user_profile`, `ability_score` |
| **Python常量** | UPPER_SNAKE_CASE | `DIMENSION_WEIGHTS`, `MAX_SCORE` |
| **数据库表** | snake_case，复数 | `users`, `ability_profiles` |
| **API端点** | kebab-case，名词复数 | `/api/v1/task-records` |
| **环境变量** | UPPER_SNAKE_CASE | `DATABASE_URL`, `JWT_SECRET_KEY` |

### 2.2 TypeScript规范

**必须遵守：**
- 禁止使用 `any`，用 `unknown` + 类型守卫替代
- 所有函数必须声明返回类型
- 接口用 `interface`（可扩展），类型别名用 `type`（联合类型等）
- Props解构时使用 `interface Props { ... }`
- 禁止 `// @ts-ignore`，确有需要时用 `// @ts-expect-error` 并附带原因注释

```typescript
// ✅ 正确
interface UserProfile {
  id: number;
  name: string;
  role: UserRole;
}

function fetchUser(id: number): Promise<UserProfile> { ... }

// ❌ 错误
function fetchUser(id: any): any { ... }
```

**导入顺序：**
1. React相关（react, react-router-dom）
2. 第三方库（antd, zustand, axios）
3. 本地模块（@/features, @/components）
4. 类型（@/types）
5. 样式

```typescript
import { useState, useEffect } from 'react';
import { useNavigate } from 'react-router-dom';

import { Button, Form, Input } from 'antd';
import { useAuthStore } from '@/features/auth/authStore';

import type { User } from '@/types';
```

### 2.4 React组件规范

```typescript
// ✅ 正确：函数式组件 + Hooks
interface TaskCardProps {
  task: Task;
  onComplete: (taskId: number) => void;
  readonly?: boolean;
}

export function TaskCard({ task, onComplete, readonly = false }: TaskCardProps) {
  const [isExpanded, setIsExpanded] = useState(false);

  const handleComplete = useCallback(() => {
    onComplete(task.id);
  }, [task.id, onComplete]);

  return (
    <Card
      title={task.name}
      extra={!readonly && <Button onClick={handleComplete}>完成</Button>}
    >
      {isExpanded && <p>{task.description}</p>}
    </Card>
  );
}
```

- 使用 Ant Design 的 token 系统进行主题定制

## 三、Git工作流

### 3.1 分支策略

```
main ─────●────────●────────●────────── (生产就绪)
           \        \        \
            \        \        \
develop ────●──●────●──●────●──●─── (集成分支)
                  \        \
                   \        \
feature/xxx ──●──●  \  feature/yyy ──●──●
```

| 分支 | 用途 | 来源 | 目标 |
|------|------|------|------|
| `main` | 生产就绪代码 | `develop` | - |
| `develop` | 集成分支 | `feature/*` | `main` |
| `feature/*` | 功能开发 | `develop` | `develop` |
| `fix/*` | Bug修复 | `develop` 或 `main` | `develop` 或 `main` |
| `release/*` | 发布准备 | `develop` | `main` |

### 3.2 提交规范（Conventional Commits）

```
<type>(<scope>): <subject>

[body]

[footer]
```

| Type | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug修复 |
| `refactor` | 重构（不改变功能） |
| `docs` | 文档变更 |
| `style` | 格式变更（空格、分号等） |
| `test` | 测试相关 |
| `chore` | 构建/工具变更 |

**示例：**
```
feat(profile): 实现能力画像历史快照功能

- 新增 ProfileSnapshot 模型
- 添加画像版本对比API
- 前端增加历史趋势图表

Closes #42
```

### 3.3 Pull Request规范

每个PR必须满足：
1. **标题**符合Conventional Commits
2. **描述**包含：做了什么、为什么做、如何测试
3. **关联Issue**（如有）
4. **CI通过**：lint + typecheck + test全部绿色
5. **至少1人Review通过**

**PR模板：**
```markdown
## 变更说明
<!-- 简述此PR做了什么 -->

## 变更原因
<!-- 为什么需要此变更 -->

## 测试方式
<!-- 如何验证此变更 -->
- [ ] 单元测试通过
- [ ] 手动测试步骤：...

## 截图（如有UI变更）
<!-- 前后对比截图 -->
```

### 4.1 测试策略

| 层级 | 工具 | 覆盖目标 | 执行频率 |
|------|------|---------|---------|
| 单元测试 | Jest / Pytest | 80% | 每次push |

## 五、代码审查标准

### 5.1 审查清单

| 类别 | 检查项 |
|------|--------|
| **安全** | 密码是否哈希？是否有SQL注入风险？敏感数据是否脱敏？ |
| **正确性** | 边界条件是否处理？空值是否处理？异步错误是否捕获？ |
| **性能** | 是否有N+1查询？useEffect是否有依赖缺失？是否有不必要的重渲染？ |
| **可维护性** | 命名是否清晰？函数是否过长（>50行）？是否有硬编码？ |
| **测试** | 是否包含对应测试？测试是否覆盖了核心路径？ |

### 5.2 审查礼仪
- 评论使用"建议"语气而非命令
- 区分"必须修改"和"建议优化"
- 每条评论附带具体改进建议

## 六、API设计规范

### 6.1 REST规范

```
GET    /api/v1/users              # 列表（支持分页、筛选）
GET    /api/v1/users/{id}         # 详情
POST   /api/v1/users              # 创建
PUT    /api/v1/users/{id}         # 全量更新
PATCH  /api/v1/users/{id}         # 部分更新
DELETE /api/v1/users/{id}         # 删除
```

### 6.2 响应格式

```json
// 成功
{
  "data": { ... },
  "meta": {
    "page": 1,
    "page_size": 20,
    "total": 150
  }
}

// 列表
{
  "data": [ ... ],
  "meta": {
    "page": 1,
    "page_size": 20,
    "total": 150
  }
}

// 错误
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "输入数据不合法",
    "details": [
      { "field": "email", "message": "邮箱格式不正确" }
    ]
  }
}
```

### 6.3 状态码使用

| 状态码 | 场景 |
|--------|------|
| 200 | 查询成功 |
| 201 | 创建成功 |
| 204 | 删除成功（无响应体） |
| 400 | 请求参数错误 |
| 401 | 未认证 |
| 403 | 无权限 |
| 404 | 资源不存在 |
| 409 | 数据冲突（如重复邮箱） |
| 422 | 验证错误 |
| 500 | 服务器内部错误 |

## 七、数据库规范

### 7.1 表命名

- 表名：snake_case，复数形式
- 关联表：`{表A}_{表B}`，按字母排序

### 7.2 字段规范

| 元素 | 规范 | 示例 |
|------|------|------|
| 主键 | `id`，UUID或自增整数 | `id = Column(Integer, primary_key=True)` |
| 外键 | `{关联表}_id` | `user_id`, `task_id` |
| 时间戳 | `created_at`, `updated_at` | `DateTime, default=utcnow` |
| 布尔值 | `is_{描述}` | `is_active`, `is_deleted` |
| 状态 | `status`，用枚举字符串 | `status = "active" \| "archived"` |

### 7.3 迁移规范

- 使用Alembic管理所有数据库变更
- 每次模型变更必须生成对应迁移文件
- 迁移文件必须可正向和反向执行
- 禁止手动修改已提交的迁移文件

## 十、常用命令速查

```makefile
# Makefile
.PHONY: dev install test lint clean

# 启动开发环境
dev:
	docker-compose up -d postgres redis
	cd backend && uvicorn app.main:app --reload &
	cd frontend && npm run dev

# 安装依赖
install:
	cd backend && pip install -r requirements.txt
	cd frontend && npm install

# 运行测试
test:
	cd backend && pytest
	cd frontend && npm test

# 代码检查
lint:
	cd backend && ruff check .
	cd frontend && npm run lint

# 类型检查
typecheck:
	cd backend && mypy app/
	cd frontend && npm run typecheck

# 数据库迁移
migrate:
	cd backend && alembic upgrade head

# 清理
clean:
	find . -type d -name __pycache__ -exec rm -rf {} +
	find . -type d -name node_modules -exec rm -rf {} +
```
