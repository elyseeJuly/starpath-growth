## 二、技术栈选型

### 2.1 选型原则

1. **渐进式复杂度**：v2.0先建立可用闭环，再逐步引入AI
2. **类型安全优先**：全栈TypeScript + Python类型注解
3. **约定优于配置**：统一的目录结构和命名规范
4. **可测试性**：每个模块独立可测

### 2.2 详细选型

| 层级 | 技术 | 版本 | 选型理由 |
|------|------|------|---------|
| **前端框架** | React | 18.3+ | 社区成熟、组件生态丰富 |
| **构建工具** | Vite | 5.x | 替代CRA，启动快、HMR优秀 |
| **UI组件库** | Ant Design | 5.x | 更适合中文场景，图标API稳定 |
| **状态管理** | Zustand | 4.x | 比Redux/Context更轻量，TS友好 |
| **路由** | React Router | 6.x | 标准选择 |
| **图表** | Recharts | 2.x | React原生，比Chart.js更符合组件化 |
| **HTTP客户端** | Axios | 1.x | 拦截器、请求取消、错误处理优于fetch |
| **表单** | React Hook Form | 7.x | 性能最优，验证集成方便 |
| **语音** | Web Speech API | 原生 | 引导系统的语音提示 |
| **后端框架** | FastAPI | 0.110+ | 异步支持、自动文档、Pydantic v2 |
| **ORM** | SQLAlchemy | 2.0+ | Python生态标准 |
| **认证** | python-jose + passlib | latest | JWT生成/验证 |
| **数据库** | PostgreSQL | 15+ | 生产级（SQLite作为开发/测试备选） |
| **缓存** | Redis | 7+ | 会话、高频数据缓存 |
| **任务队列** | Celery + Redis | latest | 异步画像更新、报告生成 |
| **测试前端** | Jest + Testing Library | latest | React组件测试 |
| **测试后端** | Pytest + httpx | latest | API测试 |
| **E2E测试** | Playwright | latest | 跨浏览器端到端测试 |
| **CI/CD** | GitHub Actions | latest | 自动构建、测试、部署 |
| **容器化** | Docker + Compose | latest | 开发环境一致性 |

### 2.3 不选的技术及原因

| 技术 | 为何不选 |
|------|---------|
| Next.js | 本项目是SPA，不需要SSR/SSG，React+Vite更轻 |
| GraphQL | REST已足够，当前阶段不需要额外复杂度 |
| MongoDB | 数据关系明确，关系型数据库更适合 |
| Kubernetes | v2.0阶段Docker Compose即可，v3.0再考虑 |
| LangChain | 多智能体先以规则引擎实现，v3.0再引入AI |

### 4.2 画像计算引擎

```
输入数据                   处理流程                    输出
──────────────────────────────────────────────────────────

行为观察      ──→  (0.4×行为 +                     │
生理指标      ──→   0.3×任务 +  →  置信度评估 →      ├─ 置信度报告
主观报告      ──→   0.2×情绪 +                      │   (数据充分度)
               0.1×生理)       →  趋势分析    →      ├─ 变化趋势
                                                     │   (30天曲线)
历史画像      ──→  版本管理     →  快照存储   →      └─ 预测曲线
```

**加权算法伪代码：**
```python
def calculate_dimension(dimension: str, data_sources: dict) -> Tuple[float, float]:
    weights = DimensionWeights.get(dimension, DEFAULT_WEIGHTS)
    score = sum(data.value * weights[data.source] for data in data_sources)
    confidence = calculate_confidence(data_sources)
    return score, confidence
```

### 4.5 Vue对比：为何选择React

| 维度 | React + Ant Design | Vue + Element Plus |
|------|-------------------|-------------------|
| TypeScript支持 | ★★★★★ 一流 | ★★★★☆ 良好 |
| 社区生态 | ★★★★★ 最大 | ★★★★☆ 大 |
| 无障碍支持 | Ant Design内置 | Element Plus有限 |
| 招聘难度 | 容易 | 国内容易 |
| 团队经验 | 已有React代码基础 | 需要重新学习 |

**结论**：保留React技术栈，从CRA迁移到Vite，从Material-UI切换到Ant Design。

## 五、数据流设计

### 5.1 核心数据流

```
用户操作（前端）
    │
    ├── 登录 → POST /api/v1/auth/login → 返回JWT → 存储localStorage
    │
    ├── 查看画像 → GET /api/v1/profiles/{id} → 返回画像+历史
    │
    ├── 提交任务记录 → POST /api/v1/tasks/record
    │       │
    │       └── 后端触发智能体链：
    │              Evaluator → Profile Update
    │              Intervenor → Generate Suggestions
    │              Reflector  → Async Strategy Review
    │
    ├── 查看建议 → GET /api/v1/profiles/{id}/suggestions
    │
    └── 场景模拟 → POST /api/v1/simulations
            │
            └── 返回模拟步骤历史 + 能力变化预测
```

### 5.2 API版本策略

- 使用URL前缀版本控制：`/api/v1/`
- v1为当前重构版本，v2预留
- 版本间向后兼容至少一个大版本
